
For the SonarQube migration, we did not define a single migration strategy. Each entity is free to choose the approach that best fits its needs.

Most entities have already migrated their projects to our  SonarQube instance, but they did it without keeping the project history.
“As a result, projects have lost their historical analysis data, especially on branches. This means that, on the first scan in the new instance, SonarQube will treat all existing code as ‘new code’. SonarQube offers multiple strategies to define ‘new code’, and projects will be able to select the most appropriate strategy to handle this situation.

“To support you better, we need to understand your specific requirements for SonarQube migration and history management. Please create a support ticket describing your needs, so that we can analyze them and propose appropriate solutions.”
